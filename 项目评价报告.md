## 二、优点

### 1. 架构设计合理
```
seckill-common (公共层) → seckill-api (接口层) → seckill-service (业务层) → seckill-web (启动层)
```
四层模块拆分清晰，依赖方向正确（外层依赖内层），符合分层架构最佳实践。

### 2. 秒杀核心流程设计正确
- **Redis 预扣库存 + MQ 异步落库** 的双层架构是业界标准做法
- Lua 脚本保证 Redis 侧库存扣减的原子性
- 分布式锁保证一人一单
- 滑动窗口限流防止刷单
- MQ 削峰填谷，快速响应用户

### 3. 异常回滚机制完整
`SeckillServiceImpl.executeSeckill()` 在 catch 块中回滚 Redis 库存和用户下单标记，finally 中释放锁，异常处理链完整。

### 4. 幂等性设计到位
MQ 消费者通过 `orderNo` 查重，防止重复消费导致重复创建订单。

### 5. 定时任务兜底
- `SeckillScheduler` 每 30s 更新商品状态
- `OrderServiceImpl.handleExpiredOrders()` 每 60s 处理超时订单并回滚库存

### 6. 代码可读性好
- Lombok 简化样板代码
- 日志级别使用恰当（info/warn/debug/error）
- 方法职责单一、命名清晰

## 三、重要缺陷（🔴 必须修复）

### 1. 分布式锁释放存在竞态条件

```java
// DistributedLockService.java:66-78 — 当前实现（非原子）
public boolean releaseLock(String key, String value) {
    String currentValue = redisTemplate.opsForValue().get(lockKey);  // 步骤1: GET
    if (value.equals(currentValue)) {                                 // 步骤2: 比较
        redisTemplate.delete(lockKey);                                // 步骤3: DEL
        ...
    }
}
```

**问题**：GET 和 DEL 之间不是原子的。如果锁在 GET 之后、DEL 之前过期并被其他线程获取，当前线程会误删别人的锁。

**修复**：使用 Lua 脚本保证原子性：
```lua
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

### 2. 限流实现非原子，存在竞态条件

```java
// RateLimitService.java:51-70
redisTemplate.execute((connection) -> {
    redisTemplate.opsForZSet().removeRangeByScore(...);  // 非原子
    Long count = redisTemplate.opsForZSet().zCard(key);   // 非原子
    ...
    redisTemplate.opsForZSet().add(...);                   // 非原子
}, true);
```

**问题**：`execute` 传入 `true`（exposeConnection）并不能提供事务原子性。多个并发请求可能同时通过限流检查，导致实际限流失效。

**修复**：所有 ZSET 操作应封装到一个 Lua 脚本中原子执行。

### 3. 密码使用 MD5 加密

```java
// Md5Util.java — 使用固定盐值 + MD5
private static final String SALT = "seckill_salt_2024";
```

MD5 已被证明不安全，配合固定盐值更是容易被彩虹表破解。应使用 **BCrypt** 或 **Argon2**。

### 4. JWT 密钥硬编码

```java
// JwtUtil.java
private static final String SECRET_KEY = "seckillSecretKey2024ForJWTTokenGeneration";
```

密钥硬编码在源码中，一旦代码泄露则所有 Token 可被伪造。应通过环境变量或配置中心注入。

### 5. 用户下单标记永不过期导致 Redis 内存泄漏

```java
// UserOrderCheckService.java:35-38
public void markOrdered(Long userId, Long goodsId, String orderNo) {
    redisTemplate.opsForValue().set(key, orderNo);  // 无过期时间！
}
```

用户成功秒杀并支付后，`seckill:user:order:{userId}:{goodsId}` key 永不过期。随着秒杀活动增多，Redis 内存会持续增长。应设置合理的过期时间（如 7 天）。

## 四、中等问题（🟡 建议修复）

### 6. 事务消息监听器是摆设

```java
// SeckillTransactionListener.java
public RocketMQLocalTransactionState executeLocalTransaction(...) {
    return RocketMQLocalTransactionState.COMMIT;  // 永远 COMMIT
}
public RocketMQLocalTransactionState checkLocalTransaction(...) {
    return RocketMQLocalTransactionState.COMMIT;  // 永远 COMMIT
}
```

`executeLocalTransaction` 没有任何业务逻辑，`checkLocalTransaction` 也不做任何回查。整个事务消息机制形同虚设——如果 MQ 消费失败，没有事务回查来保证最终一致性。**要么实现真正的事务回查，要么移除这个类改用普通消息。**

### 7. catch 块重复代码

```java
// SeckillServiceImpl.java:94-108
} catch (SeckillException e) {
    redisStockService.increaseStock(goodsId);       // 重复
    userOrderCheckService.removeOrderMark(userId, goodsId);  // 重复
    throw e;
} catch (Exception e) {
    redisStockService.increaseStock(goodsId);       // 重复
    userOrderCheckService.removeOrderMark(userId, goodsId);  // 重复
    ...
}
```

两个 catch 块的回滚逻辑完全相同，应提取为私有方法 `rollbackRedisOperations()`。

### 8. 商品校验每次都查数据库

```java
// SeckillServiceImpl.java:114-116
private SeckillGoods validateGoods(Long goodsId) {
    SeckillGoods goods = goodsService.getGoodsById(goodsId);  // 每次都查MySQL
```

高并发下每个秒杀请求都查 MySQL 校验商品状态，数据库会成为瓶颈。商品的状态信息应缓存到 Redis。

### 9. 库存 Key 未设置过期时间

```java
// RedisStockService.java:50-53
public void initStock(Long goodsId, Integer stock) {
    redisTemplate.opsForValue().set(key, String.valueOf(stock));  // 无TTL
}
```

仅在 `SeckillScheduler` 商品结束时才删除。如果商品状态更新失败或系统崩溃，key 会残留。

### 10. 秒杀结果轮询接口未实现

README 中提到了 `GET /api/seckill/result/{orderNo}` 接口，但实际代码中没有实现对应的 Controller 方法。用户秒杀后无法通过轮询得知最终结果。

### 11. 配置类冗余

`seckill-common` 中的 `RedisConfig.java` 和 `WebConfig.java` 实际上不会被 Spring 扫描到（扫描路径在 web 模块），真正的配置在 `seckill-web` 模块中。两个相同的 CORS 配置、两个相同的 Redis 配置，容易造成维护混乱。

### 12. Lua 脚本内联为 Java 字符串

```java
private static final String DECREASE_STOCK_SCRIPT =
    "if redis.call('exists', KEYS[1]) == 1 then " +
    "local stock = tonumber(redis.call('get', KEYS[1])); " + ...
```

作为独立 `.lua` 文件管理更利于测试和维护，也能利用 IDE 的 Lua 语法高亮。

### 13. 每次调用都 new DefaultRedisScript

```java
// RedisStockService.java:71
DefaultRedisScript<Long> script = new DefaultRedisScript<>(DECREASE_STOCK_SCRIPT, Long.class);
```

每次扣库存都创建一个新的 `DefaultRedisScript` 对象。应改为类级别的 static final 常量以复用。

## 五、改进建议

| 优先级 | 问题 | 建议方案 |
|--------|------|----------|
| P0 | 分布式锁释放非原子 | 改用 Lua 脚本实现 `releaseLock` |
| P0 | 限流非原子 | 限流逻辑改为 Lua 脚本 |
| P0 | MD5 密码 | 迁移到 BCrypt |
| P1 | JWT 密钥硬编码 | 改为环境变量注入 |
| P1 | 用户标记无过期 | 设置 7 天 TTL |
| P1 | 事务消息空壳 | 实现真正的事务回查或移除 |
| P2 | 重复 catch 块 | 提取公共回滚方法 |
| P2 | 每次都查 DB | 商品状态缓存到 Redis |
| P2 | Lua 脚本内联 | 提取为独立 .lua 文件 |
| P2 | DefaultRedisScript 重复创建 | 改为 static final |
| P3 | 配置类冗余 | 清理无用的 common 配置类 |
| P3 | 轮询接口缺失 | 实现订单结果查询 |

---

## 六、总结

这个项目**骨架搭建得很好**——架构分层、核心流程、技术选型都是正确的方向。代码注释充分、命名规范，可读性不错。

但是，**高并发场景下的正确性依赖细节**。分布式锁释放的非原子操作、限流的竞态条件，这两个问题在生产环境的极端并发下会导致**超卖**和**限流失效**——而这恰恰是秒杀系统最不能出错的环节。

一句话总结：**这是一个优秀的课程设计/原型项目，但距离生产环境还有一段距离，至少需要修复上述 P0/P1 问题后才能上线。**
